gstack 互动拆解课
1

产品方法论

gstack 不是“23 个命令合集”，而是一条把创意、评审、实现、测试、发布串成单一冲刺流水线的方法论系统。

为什么它不是普通命令集合

CODE

/office-hours
/plan-ceo-review
/plan-eng-review
/review
/qa
/ship
PLAIN ENGLISH

这组命令不是平铺摆放的工具箱，而是按冲刺顺序排列的角色链。

`/office-hours` 先把模糊需求改写成真正的问题定义，`/plan-*` 再把产品和工程前提锁死，后面的 `/review`、`/qa`、`/ship` 才不是盲修盲测。

02

角色协同与团队模式

gstack 的关键不是“谁会写代码”，而是每个角色都消费上游产物并生成下游输入，像真实团队一样交接上下文。

群聊动画

OFF
/office-hours

我先追问真实痛点，把“想做的功能”改写成“真正的产品工作”。

CEO
/plan-ceo-review

我挑战范围和方向，找出这个需求背后更像 10-star 产品的版本。

ENG
/plan-eng-review

我把愿景落到架构、状态机、失败路径和测试矩阵，防止实现阶段靠猜。

QA
/qa

我用真实浏览器回归验证，并把 bug 修复与回归测试一起补上。

团队模式安装片段

CODE

git clone --single-branch --depth 1 https://github.com/garrytan/gstack.git ~/.claude/skills/gstack
cd ~/.claude/skills/gstack && ./setup --team
~/.claude/skills/gstack/bin/gstack-team-init required
git add .claude/ CLAUDE.md
PLAIN ENGLISH

这里体现的是“全员共享方法论”而不是把一堆提示词 vendoring 进仓库。

个人全局安装负责自动升级，项目内只保留最薄的一层接线配置，让团队里的 Claude Code 会话默认按同一套流程协作。

03

浏览器底座与执行链路

gstack 的方法论之所以能成立，是因为它不只是一堆 Markdown 技能；它还有一个低延迟、可持久化的浏览器运行时来支撑 QA、设计审查和网页调试。

消息流

🤖
Claude Code
🧰
CLI Binary
⚡
Bun Server
🌐
Chromium
点击 Next Step 开始

为什么这个底座决定了课程价值

CODE

{ "pid": 12345, "port": 34567, "token": "uuid-v4", "binaryVersion": "abc123" }
Bun.serve()
Authorization: Bearer <token>
First call ~3s, later calls ~100-200ms
PLAIN ENGLISH

`browse.json` 是技能与浏览器之间的接线板，记录当前 daemon 的 PID、端口和鉴权令牌。

Bun 服务常驻后，后续 QA 与浏览器操作不必反复冷启动；这正是 gstack 能把“计划审查”和“真实浏览器验证”连成闭环的底层原因。

4

文档生成与安全治理

gstack 的另一条底层主线是“让技能说明、评审门禁和高风险保护持续与代码同步”，否则整套 AI 工厂会很快漂移失真。

代码与规则片段

CODE

SKILL.md.tmpl -> gen-skill-docs.ts -> SKILL.md
Review Readiness Dashboard
/careful
/freeze
/guard
PLAIN ENGLISH

模板系统负责避免技能文档和真实命令漂移，Dashboard 负责告诉你哪些审查已跑、哪些还欠账。

`/careful`、`/freeze`、`/guard` 则是在 AI 编码变快以后，用来约束误删、误改和越界写入的最后一道保险丝。
